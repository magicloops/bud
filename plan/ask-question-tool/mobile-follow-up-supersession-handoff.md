# Mobile Handoff: Follow-Up Messages Supersede Ask User Questions

## Status

- Feature: normal follow-up messages while `ask_user_questions` is pending
- Service behavior: implemented
- Reference web behavior: implemented
- Related handoff: [mobile-client-handoff.md](./mobile-client-handoff.md)

This document focuses only on the follow-up-message path. The structured prompt render/answer contract remains covered by the general mobile handoff.

## Behavior Summary

When a thread is paused on an `ask_user_questions` prompt, mobile should still let the user send a normal follow-up message from the composer.

That follow-up send does not submit an `ask_user_questions_response_v1` payload. Instead, mobile sends the normal message request:

```text
POST /api/threads/:thread_id/messages
Content-Type: application/json
```

The service then:

1. closes all pending `ask_user_questions` requests for the thread as skipped
2. records a skipped `ask_user_questions` tool result for the old waiting turn when it can
3. emits a successful `final` event for the old turn with `reason: "superseded_by_user_message"`
4. persists the new user message
5. starts a fresh agent turn from the new message

This is intentionally different from tapping Skip all on the prompt card. Explicit Skip all continues the original turn with skipped answers. A normal follow-up message stops the old waiting turn and starts from the user's new text.

## When To Enable The Composer

If `/agent/state` says:

```json
{
  "active": true,
  "phase": "waiting_for_user",
  "pending_tool": {
    "name": "ask_user_questions"
  }
}
```

or the live stream has shown an `agent.tool_call` with `name: "ask_user_questions"`, keep the normal message composer enabled.

Do not show a global "agent still working" spinner for this state. The agent is waiting on user input. The pending question card can remain visible, but the user should be able to ignore it and send text.

## Message Send Request

Use the same request shape as any normal user message:

```json
{
  "text": "Actually, skip deployment for now and just update the README.",
  "client_id": "018f4f2a-8b6a-7c10-9234-2b5cc27f62c0",
  "model": "gpt-5.5",
  "reasoning_effort": "low"
}
```

Notes:

- `client_id` should be generated by the mobile client as a UUID.
- `model` and `reasoning_effort` follow the same rules as other message sends.
- Do not include question answers in this request.
- Do not call the question-response route before sending the message.

## Message Send Response

The create-message response includes the canonical serialized user message:

```json
{
  "message_id": "22222222-2222-4222-8222-222222222222",
  "client_id": "018f4f2a-8b6a-7c10-9234-2b5cc27f62c0",
  "message": {
    "message_id": "22222222-2222-4222-8222-222222222222",
    "client_id": "018f4f2a-8b6a-7c10-9234-2b5cc27f62c0",
    "role": "user",
    "display_role": "User",
    "content": "Actually, skip deployment for now and just update the README.",
    "metadata": {
      "model": "gpt-5.5",
      "reasoning_effort": "low",
      "model_selection_source": "explicit_request"
    },
    "created_at": "2026-05-21T18:30:05.000Z"
  }
}
```

Mobile should replace its optimistic row with `response.message`, not just patch ids. The canonical `created_at` is important because the skipped tool-result row is persisted before the follow-up user message. Preserving a local optimistic timestamp can make the live feed order differ from refresh order.

Recommended timeline sort:

1. `created_at`
2. `message_id`
3. `client_id` only as a fallback for synthetic or optimistic rows

## Stream Events To Expect

The old waiting turn may emit an `agent.tool_result` for the skipped questions:

```json
{
  "turn_id": "01OLDTURN...",
  "client_id": "33333333-3333-4333-8333-333333333333",
  "call_id": "call_questions",
  "name": "ask_user_questions",
  "summary": "Skipped questions: Deployment details",
  "user_questions": {
    "kind": "user_questions",
    "requestId": "qr_01J...",
    "responses": [
      {
        "question_id": "target_environment",
        "status": "skipped",
        "skip_reason": "user_skipped"
      }
    ]
  },
  "message": {
    "message_id": "44444444-4444-4444-8444-444444444444",
    "client_id": "33333333-3333-4333-8333-333333333333",
    "role": "tool",
    "display_role": "Tool",
    "content": "{\"tool\":\"ask_user_questions\",\"result\":{\"schema\":\"ask_user_questions_tool_result_v1\",\"responses\":[]}}",
    "metadata": {
      "tool": "ask_user_questions",
      "call_id": "call_questions",
      "kind": "user_questions"
    },
    "created_at": "2026-05-21T18:30:04.000Z"
  }
}
```

Then the old turn may emit:

```json
{
  "turn_id": "01OLDTURN...",
  "status": "succeeded",
  "reason": "superseded_by_user_message"
}
```

Treat this final event as terminal for the old prompt. It may not include assistant `text` or `message_id`.

The new follow-up turn then proceeds through the normal stream events for the new user message.

## Client State Reconciliation

Recommended mobile flow:

1. Render the pending `ask_user_questions` prompt from `agent.tool_call` or `/agent/state`.
2. Keep the normal composer enabled while the prompt is pending.
3. If the user sends a follow-up message, create an optimistic user row keyed by the generated `client_id`.
4. POST the normal message request to `/api/threads/:thread_id/messages`.
5. When the response returns, replace the optimistic row with `response.message`.
6. Keep the agent stream attached. If no stream is attached, fetch `/agent/state` after the message response.
7. On `agent.tool_result` with `name: "ask_user_questions"`, replace the pending prompt row with `event.data.message` when present.
8. On `final.reason === "superseded_by_user_message"`, clear any remaining pending prompt UI for that `turn_id` and do not show an error.
9. On `agent.resync_required`, refetch `/messages` and `/agent/state`, then reattach the stream.

If the POST fails, remove or mark the optimistic user row as failed and leave the pending prompt visible. The service did not confirm the supersession.

## Idempotency And Races

- The message route checks duplicate `client_id` before superseding pending questions. A retry with the same `client_id` returns the existing message and should not duplicate supersession side effects.
- If an explicit prompt response and a follow-up message race, one wins. The losing client path should refresh `/messages` and `/agent/state`.
- If explicit answer wins first, the old turn may continue normally.
- If follow-up supersession wins first, the explicit answer submit may receive an already-answered or not-pending response. Treat that prompt as stale and refresh.

## Error Handling

| Status | Client Behavior |
|--------|-----------------|
| `201` / `200` | Replace optimistic row with `response.message`; keep or attach stream. |
| `400` | Show the message-send validation error; keep the prompt visible. |
| `401` | Follow normal mobile auth recovery. |
| `404` | Treat the thread as inaccessible/stale and refresh navigation state. |
| `409` | Client id conflict; generate a new `client_id` only if this was not a retry of the same local send. |
| `5xx` | Keep prompt visible and offer retry. |

## Attention And Notifications

The current shipped behavior does not yet define durable `human_input_requested` attention resolution or stale-push suppression for superseded prompts. Mobile can locally dismiss pending prompt UI after successful follow-up supersession, but should not assume notification-state semantics beyond the thread/message stream state until Phase 9 lands.

## Mobile Test Checklist

- Composer stays enabled while `phase: "waiting_for_user"`.
- Sending a normal follow-up does not call the question-response route.
- Optimistic user row is replaced with `response.message`.
- Skipped `ask_user_questions` tool result appears before the follow-up user message in live order and after refresh.
- `final.reason === "superseded_by_user_message"` clears the pending prompt without showing an assistant-empty error.
- Retrying the same message `client_id` does not duplicate prompt closeout.
- Race between explicit answer and follow-up send resolves through refresh without duplicate visible prompts.
- Offline/reconnect path refetches `/messages` plus `/agent/state` and converges to persisted order.
